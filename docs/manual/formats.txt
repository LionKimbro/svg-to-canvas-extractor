
  == ﻿Formats Used by svg2canvasx ==


  Inkscape SVG

  This is the authoring format and the normal human-edited source for the
  project.  The SVG is expected to come from Inkscape and to follow
  svg2canvasx's layer and labeling conventions, especially explicit
  bracket-tagged layer roles such as [visual], [annotate], and [comment].  At
  this stage the file still contains full SVG structure, local coordinates,
  transforms, group nesting, presentation styles, text spans, and authoring
  metadata.  svg2canvasx does not treat this as the final runtime format; it
  treats it as the structured drawing source that must be normalized before
  later code can reason about it safely.


  Expanded JSON
  
  This is the Stage 1 extraction format produced by the extract command.  It
  is a normalized, verbose JSON description of the selected SVG content.  It
  preserves source metadata, SVG size information, layers, extracted drawable
  objects, annotations, styles, raw path data, labels, tags, local geometry,
  and transformed world geometry when available.  The expanded JSON is the
  main diagnostic and interchange format for downstream tooling because it
  keeps enough detail to preview the drawing, inspect object placement, and
  perform later semantic analysis without having to reopen the original SVG.


  Flow JSON

  This is a more compact, process-oriented JSON derived from the expanded
  extraction JSON.  It groups data by logical layer and separates presentation
  items from semantic regions in a form that is easier for later layout,
  preview, generation, or LLM consumption.  In a flow file, presentation
  layers hold drawable items, while annotation information is represented as
  compact regions attached to layers.  Compared with the expanded JSON, the
  flow format intentionally throws away much of the raw SVG detail and keeps
  the information most useful for downstream structured processing.


  Compiled Python
  
  This is the generated Python source produced from a flow JSON file by the
  rawtkinter command.  The output is not a full application; it is a code
  artifact that contains drawing logic suitable for use with an existing
  Tkinter Canvas.  In practice, it is a compiled or lowered representation of
  the flow document into imperative Python drawing calls.  This format is
  useful when a project wants a lightweight, inspectable, code-level rendering
  artifact instead of shipping JSON plus a renderer at runtime.
